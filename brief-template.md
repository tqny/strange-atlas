# Project Brief Template

Use this format to define a new portfolio project. You can fill it out yourself or build it up through conversation with Claude.

Not every field is required for every project — but the more specific you are upfront, the stronger the planning output will be.

---

## Project Idea

What is the product? One or two sentences.

## Purpose

Why am I building this? What role, workflow, business problem, or product scenario is this meant to help me understand, simulate, or demonstrate?

## Target User

Who is this for? Be specific about the primary user and their context. (For portfolio projects, the "user" might be a persona you're simulating — e.g., "a sales manager reviewing pipeline health.")

## Success Criteria

What would make this project feel successful as a portfolio-grade artifact? What should a reviewer or user be able to understand or do?

## MVP

What must be included in version 1? List only the must-have features. Be ruthless about scope.

## Non-Goals

What should explicitly not be built yet? What belongs in later phases? Being clear here prevents scope creep.

## Platform

What is the product format?

Examples: web app, dashboard, internal tool, mobile-first web app, landing page + app shell, data explorer, workflow simulator.

## Core User Flow

What is the main thing the user should be able to do from start to finish? Describe the primary workflow in a few sentences.

## Data / State Needs

What kind of data does the project need?

Examples: static mock data, local state, localStorage, backend database, external API, user-generated data, analytics/events.

## Auth / Access Needs

Does the MVP need login, user accounts, roles, or protected views? If yes, keep it minimal and realistic. If no, say "none for MVP."

## Design Direction

What should it feel like visually? Describe:

- Tone (professional, playful, dense, minimal, editorial, etc.)
- Density (information-heavy dashboard vs. clean workflow tool)
- Polish level (production-ready vs. functional prototype)
- UI style (dark mode, light mode, cards, tables, sidebar, etc.)
- Interaction style (static, animated, highly interactive)
- Any aesthetic references in words

## Design References

If applicable, provide:

- **Reference Set 1** (style authority) — the primary visual language to follow
- **Other reference sets** (pattern candidates) — useful patterns to consider
- Notes on what you like from each

If none, say: "No specific references. Use best judgment based on design direction."

## "About This Project" Preference

How should the in-product reviewer-facing explanation surface appear?

Examples: dedicated page, tab, modal, drawer, landing-page section.

Or say: "Choose the most natural UX form for this product."

## Technical Preferences

Any preferences for:

- Framework (React, Next.js, vanilla JS, etc.)
- Styling approach (Tailwind, CSS modules, custom properties, etc.)
- Component library
- Charting library
- Database
- Auth provider
- Deployment target (GitHub Pages, Vercel, Netlify, etc.)
- TypeScript or JavaScript
- Other tooling

If none, say: "Recommend the simplest strong setup."

## Constraints

List important constraints such as:

- Speed / timeline
- Simplicity
- Beginner-friendly implementation
- Presentability / recruiter-facing polish
- Limited infrastructure
- Low / no cost
- No backend for MVP
- Mobile responsiveness
- Accessibility requirements

## Repo / Workflow Preferences

Any preferences for:

- Branch workflow
- PR workflow
- GitHub usage
- Documentation level
- Commit style

If none, say: "Use the default V3 portfolio workflow."

## Risks / Known Challenges

What already seems tricky, ambiguous, or easy to overbuild?

## Optional Notes

Anything else Claude should know before planning.
